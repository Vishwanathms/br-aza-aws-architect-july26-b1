To allow **cross-account access in AWS IAM**, you typically need **two sides** configured:

1. **Trust policy** (on the target account role)
2. **Permission policy** (on the source account user/role)

### Example Scenario

* **Account A** = `111111111111` (source)
* **Account B** = `222222222222` (target)
* User/Role in Account A wants to access resources in Account B.

---

## Step 1: Create Role in Target Account (Account B)

Create an IAM Role in Amazon Web Services Account B.

### Trust Policy (AssumeRole Policy)

This allows Account A to assume the role.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111111111111:root"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

More restrictive (specific role):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111111111111:role/devops-role"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

---

## Step 2: Attach Permissions to That Role (Target Account)

Example: Allow S3 read.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket",
        "s3:GetObject"
      ],
      "Resource": [
        "arn:aws:s3:::my-cross-account-bucket",
        "arn:aws:s3:::my-cross-account-bucket/*"
      ]
    }
  ]
}
```

---

## Step 3: Allow Source Account to Assume Role

Attach policy to User/Role in Account A:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::222222222222:role/CrossAccountS3Role"
    }
  ]
}
```

---

## Step 4: Assume the Role

Using CLI:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::222222222222:role/CrossAccountS3Role \
  --role-session-name crossaccountsession
```

Export credentials:

```bash
export AWS_ACCESS_KEY_ID=xxx
export AWS_SECRET_ACCESS_KEY=xxx
export AWS_SESSION_TOKEN=xxx
```

## Real-world Use Cases

* Cross-account Terraform deployments
* Centralized logging account
* Shared S3 bucket access
* CI/CD pipeline deployment into multiple accounts
* EKS cluster management across accounts

### Quick Memory Trick:

* **Trust policy** = "Who can enter?"
* **Permission policy** = "What can they do?"
* **AssumeRole policy** = "Who is allowed to ask?"
